# Methodology Correctness Plan

**Created:** 2026-04-27
**Status:** ✅ COMPLETE — all 9 fixes shipped 2026-04-27.
**Owner:** Mack + assistant
**Driver:** "Yes, accuracy above all else."

This document lays out the plan to remove every known methodology
flaw that produces look-ahead bias or in-sample optimization in the
QuantOpsAI system. It exists because we discovered (2026-04-27) that
the meta-model's reported AUCs of 0.83-0.96 were a data-leakage
artifact from a random train/test split, and a follow-up audit of
the broader system surfaced six more issues of the same family. The
user's explicit instruction was "we need to do it all."

The pattern is the same across the surfaced issues: **looking at
data the model wasn't supposed to have access to at the moment of
its prediction**. Once the engine for "what data exists at time T"
is wrong, every higher-level statistical test (walk-forward, OOS,
self-tuning, decay detection) inherits the bug.

*calling* the foundation correctly. Wave 3 cannot start until the
foundation produces clean data, since all three consume metrics
generated by lower waves.

## 3. Per-fix execution plan

Every fix follows the same shape:

1. **Implementation** — minimal correct change.
2. **Anti-regression test** — structural (AST/regex on source) AND
   behavioral where possible.
3. **Migration** — what to do with stale data on prod (delete a pkl,
   re-resolve labels, etc.).
4. **CHANGELOG entry** — honest framing including expected impact
   on observable metrics.
5. **Deploy** through hardened sync.sh.

### Fix #2 — `backtest_strategy` accepts date ranges

**Goal:** Replace `days=N` (today-relative) with explicit `start_date`,
`end_date` parameters. Internally still call yfinance / Alpaca with
those dates.

**Backwards compat:** Keep `days=N` accepted as a deprecated alias
that sets `end_date = today` and `start_date = today - N`. All
callers eventually migrated.

**Anti-regression:** AST test asserting `walk_forward_analysis` and
`out_of_sample_degradation` no longer pass `days=` — only `start_date`
/ `end_date`. Behavioral test that two backtests with disjoint date
ranges read disjoint data.

**Migration:** none — function-level signature change.

### Fix #6 — Forward-bar resolution at fixed horizon

**Goal:** Predictions resolve at price[T + N_bars] not "current price."
Default N = 5 trading bars (~1 trading week) for daily-resolution
profiles.

**Schema:** `ai_predictions` already has `resolution_price` and
`resolved_at`. Add `resolution_horizon_bars INTEGER DEFAULT 5` so
each prediction records the horizon it was evaluated at.

**Anti-regression:** Behavioral test seeds 10 predictions with known
prices, advances time, runs `_resolve_one`, asserts each used
`price[T + 5]` not `price[T]`. Structural test that
`_get_current_price` is NOT called from the resolution path.

**Migration:** Existing resolved predictions keep their labels (we
don't rewrite history). New predictions resolve correctly. The
meta-model retrain will mostly use new clean labels as they
accumulate.

### Fix #3 — Walk-forward really walks forward

**Goal:** Fold k uses `[start + k*W, start + (k+1)*W]` where `start`
is `today - history_days` and `W = history_days / folds`. Each fold
reads disjoint data.

**Depends on:** #2.

**Anti-regression:** Behavioral test mocking `backtest_strategy` to
record the (`start_date`, `end_date`) of each call; asserts no two
folds overlap.

### Fix #4 — Out-of-sample window doesn't overlap in-sample

**Goal:** `train_end_date = today - oos_days`. `IS` runs
`(today - history_days, train_end_date)`. `OOS` runs
`(train_end_date, today)`. Strict separation.

**Depends on:** #2.

**Anti-regression:** Behavioral test asserting `IS.end_date <
OOS.start_date` always.

### Fix #5 — Self-tuner train/validate split

**Goal:** Adjustments are computed on resolved predictions ≤ T-N days
ago (the *adjustment window*), then the *expected effect* of the
adjustment is measured on resolved predictions in (T-N, T] (the
*validation window*). Adjustments are only applied if the validation
window's metric agrees.

**Implementation sketch:**
```
N = 14 days (validation window length, configurable)
adjustment_window = predictions resolved ≤ T-N
validation_window = predictions resolved in (T-N, T]
proposed = compute_adjustment(adjustment_window)
if metric(validation_window | proposed) ≥ metric(validation_window | current):
    apply
```

**Anti-regression:** Behavioral test seeds predictions in the two
windows where the adjustment is right for the adjustment window but
wrong for the validation window; asserts the tuner does NOT apply
the change.

**Migration:** Adds `validation_window_days` field to user/profile
config. Default 14.

### Fix #8 — Alpha-decay window discipline

**Goal:** When computing "rolling Sharpe at snapshot time T," the
window is strictly `[T - 30, T)` — does NOT include T or any data
resolved after T. Lifetime baseline is `[T - history, T - 30]` —
strictly before the rolling window.

**Depends on:** #6 (because resolution timestamp matters for the
window boundary).

**Anti-regression:** Behavioral test that
`compute_rolling_metrics(as_of=T)` returns identical metrics
regardless of what data was resolved after T.

### Fix #7 — Strategy lifecycle inherits real gates

**Auto-fixed by:** #3 + #4. The lifecycle just calls
`validate_strategy()`. Once the inner gates are real, lifecycle
correctness follows. Add only a contract test that
`validate_and_promote` actually calls `validate_strategy` with a
date range, not a relative window.

### Fix #9 — Specialist confidence calibration

**Goal:** Per-specialist Platt-scaling layer. For each specialist S,
fit a logistic regression mapping raw confidence (0-100) to actual
P(correct) using resolved predictions where the specialist's
verdict was the dominant input.

**Depends on:** #5 + #6 (need clean labels on a meaningful
horizon to fit the calibration).

**Implementation:** New module `specialist_calibration.py` with
`fit_calibrators(db_path, specialists)` and `apply_calibration(raw_conf,
specialist) -> calibrated_conf`. Daily retrain task in scheduler.
Apply at the point ensemble verdicts are aggregated.

**Anti-regression:** Behavioral test seeds a specialist that's
systematically over-confident (always says 90, only right 50% of
time); asserts the calibrator maps 90 → ~50.

## 4. Migration & rollout strategy

The fixes will land in a series of commits, each:

- Self-contained: passes the full test suite on its own.
- Deployed via hardened sync.sh (pre-flight gate ensures origin/main
  matches local before each deploy).
- Followed by an explicit metric-watch period (24-72h) before
  starting the next fix.

**During Wave 1 deployment:**

The dashboard will continue reporting old-methodology metrics for
a period because:
- Meta-model AUCs already corrected (deployed `cd2d207`); next
  retrain at 3:55 PM ET fills in real numbers.
- `actual_outcome` labels already in the DB stay as-is. New
  predictions resolved post-#6 use forward-bar labels.
- Validation reports under Phase 2 will only become honest after #3
  + #4 land.

The honest framing for any partner / user looking at the dashboard
during this window: **"we're in the middle of a methodology audit;
metrics published before 2026-04-27 should be considered
provisional."** Once Wave 1 + Wave 2 land, the dashboard becomes
trustworthy in absolute terms.

**Honest expectation of impact on observable metrics:**

| Metric | Before fix | After fix (expected) |
|---|---|---|
| Meta-model AUC | 0.83 - 0.96 | **0.50 - 0.65** |
| Phase 2 walk-forward "consistency_pct" | varies | likely lower |
| Phase 2 OOS Sharpe degradation | varies | likely larger gap |
| Self-tuner adjustment frequency | high | **lower** (validation window will reject many proposed changes) |
| Alpha-decay deprecations | varies | likely **more** strategies flagged |

The system may *appear* to have less edge than the dashboard
currently suggests. That's the point — we now have honest numbers
to act on.

## 5. Anti-regression discipline

Every fix lands with a structural test that prevents the broken
pattern from re-appearing. The pattern follows the precedent set
by:

- `tests/test_meta_model_time_ordered_split.py` (just added) — AST
  + behavioral tests preventing return to random split.
- `tests/test_no_per_symbol_bars_in_web_path.py` — AST tests
  preventing per-symbol bar storms.
- `tests/test_no_snake_case_in_api_responses.py` — endpoint sweeps.

Each new structural test is named to describe what it forbids
(`test_walk_forward_uses_disjoint_date_ranges`,
`test_self_tuning_uses_validation_window`, etc.) and references this
plan in its docstring so the connection is preserved.
