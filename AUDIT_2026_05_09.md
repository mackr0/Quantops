### Issue 2 — Admin "API Calls Today" identical for every user row
**Status:** 🛠️ in progress
**Source:** page audit; `views.py:4480-4491` (actual lines `4491-4502` after re-read)
**Reported:** the route sums `total_calls` / `total_cost` across every `quantopsai_profile_*.db` and assigns the same total to every user row. No per-user attribution. Privacy/attribution issue if a real second user is ever onboarded.

**Deep-dive verification (2026-05-09):**
- Confirmed at `views.py:4491-4502`. Exact code:
  ```python

      s = spend_summary(f)
      total_calls += s["today"]["calls"]

  for u in users:
      u["api_calls_today"] = total_calls
      u["api_cost_today"]  = round(total_cost, 2)
  ```
  `glob` returns ALL profile DBs system-wide; the same total is then stamped onto every user row.
- Currently the only logged-in user is `mack@…` (admin), so the **today's user-visible damage is zero** — there's no second user to mis-attribute to. But:
  - The moment a second user account is added, the admin panel will show *the admin's* API costs against that second user (and vice versa) → privacy / attribution leak.
  - The total shown is also wrong as a "per-user" number even today: it's the system aggregate, not "how much has this user spent today."
- Schema confirms a clean per-user attribution path:
  - `trading_profiles.user_id` exists (`models.py:142`).
  - Per-profile DB filename is `quantopsai_profile_<profile_id>.db` (matches `glob` pattern).
  - Helper `models.get_user_profiles(user_id)` already returns *all* profiles for a user (enabled + disabled). Disabled profiles still have a DB on disk and their historical cost should still count toward "today" if they ran earlier in the day before being toggled off.
  - `spend_summary(db_path)` already returns zeros if the DB file is missing (`ai_cost_ledger.py:73-77`), so no extra existence check needed.
- Template `templates/admin.html:32` reads `{{ u.api_calls_today }}`. No template change required — just the data needs to be per-user.

**Plan:**
- Replace the system-wide glob in the admin route with a per-user loop:
  ```python
  for u in users:
      profiles = get_user_profiles(u["id"])
      calls = cost = 0
      for p in profiles:
          db_path = f"quantopsai_profile_{p['id']}.db"
          try:
              s = spend_summary(db_path)
              calls += s["today"]["calls"]
              cost  += s["today"]["usd"]
          except Exception as exc:
              logger.warning(
                  "admin: spend_summary failed for user=%s profile=%s db=%s: %s",
                  u["id"], p["id"], db_path, exc,
              )
      u["api_calls_today"] = calls
      u["api_cost_today"]  = round(cost, 2)
  ```
- Behavioral test: stub two users with two profiles each; assert each user's row reflects ONLY their own profiles' cost; assert no leakage between users.
- Edge-case test: a user with zero profiles → row shows 0 / $0.00 (not None, not error).
- Edge-case test: a profile DB whose `spend_summary` raises → that profile contributes 0, the rest of the user's profiles still aggregate; endpoint still returns 200.
- Cross-cutting guardrail: AST-scan for `glob.glob("quantopsai_profile_*.db")` outside `ai_cost_ledger.py` / scheduler bootstrap. Any *user-attributed* surface (admin, dashboards, per-user reports) must source from `get_user_profiles(user_id)`, not a system glob. Targeted enough to allow the legitimate scheduler-side glob (which intentionally walks all profile DBs).
- Risks: none material. The new logic always returns ≤ the old system-wide total per user; tests pin no leakage; admin template unchanged. No prod deploy concerns beyond a route change.

**Outcome (2026-05-09):**
- ✅ Commit `84499aa`. `views.admin` now loops users → `get_user_profiles(u['id'])` → sums `spend_summary` per profile owned by that user. Per-profile failures log a WARNING with user/profile/db and contribute 0; the rest of the user's profiles still aggregate.

### Issue 4 — Profit factor calc excludes most signal types
**Status:** 🛠️ in progress
**Source:** page audit; `views.py:2196` and `:3107` (actual lines `2200-2218` and `3114-3130`)
**Reported:** queries restrict to `predicted_signal IN ('BUY','SELL')` — excluding `STRONG_BUY`, `STRONG_SELL`, `SHORT`, `STRONG_SHORT`, `MULTILEG_OPEN`, `PAIR_TRADE`. Headline "Prediction Accuracy" / "Profit Factor" reflects a small subset of trades.

**Deep-dive verification (2026-05-09):**
- Confirmed in both `/performance` (views.py:2200-2218) and `/ai` (views.py:3114-3130). Identical bug shape — same `predicted_signal IN ('BUY', 'SELL')` whitelist:
  ```python
  rows = conn.execute(
      "SELECT actual_return_pct FROM ai_predictions "
      "WHERE status='resolved' AND actual_return_pct IS NOT NULL "
      "AND predicted_signal IN ('BUY', 'SELL')"
  ).fetchall()
  ```
- **Real-world impact (production data, 11 profiles):**
  | Signal | Resolved+returns | Counted? |
  |--------|-----|---|
  | BUY    | 785 | ✅ |
  | SELL   | 3   | ✅ |
  | SHORT  | 25  | ❌ |
  | MULTILEG_OPEN | 113 | ❌ |
  | HOLD   | ~13,500 | ❌ (correct — no trade) |

  → **138 of 926 actually-traded predictions (~15%) are silently dropped from profit_factor.** The headline number is computed on a subset; SHORT positions and MULTILEG_OPEN options never count toward it.
- The dropped types are *not* hypotheticals. SHORT lives on enabled small-cap-shorts profiles; MULTILEG_OPEN is the dominant signal type for all options profiles. Profit factor on options profiles is essentially meaningless today (only stock BUYs count).
- Audit also lists `STRONG_BUY/STRONG_SELL/STRONG_SHORT/PAIR_TRADE` — none of these appear in any current prod DB. Don't whitelist phantom types; just stop whitelisting.
- Convention check: `ai_tracker.py:614,620,723,730` already uses `UPPER(predicted_signal) = 'HOLD'` as the canonical "no-trade" filter. So the project-wide convention is **HOLD = no trade, everything else = trade**. The profit_factor query just doesn't follow it.

**Why missed:** the IN(...) whitelist was correct when only BUY/SELL existed. New signal types (SHORT, MULTILEG_OPEN) were added to the prediction recorder but the downstream consumer queries were never re-audited. Classic whitelist-rot. (Issue 1 was a structural mistake, this is the same lesson — **don't whitelist closed sets when an open-world exclusion expresses the intent.**)

**Plan:**
- Replace whitelist with HOLD-exclusion in both routes:
  ```python
  "WHERE status='resolved' AND actual_return_pct IS NOT NULL "
  "AND predicted_signal IS NOT NULL "
  "AND UPPER(predicted_signal) != 'HOLD'"
  ```
  This matches the established convention (ai_tracker.py uses UPPER(...)='HOLD' for the inverse filter).
- Behavioral test: seed a DB with one of each {BUY, SELL, SHORT, MULTILEG_OPEN, HOLD, NULL} predictions with known returns; assert profit_factor includes BUY+SELL+SHORT+MULTILEG_OPEN, excludes HOLD and NULL, and matches hand-computed value.
- Future-proofing test: assert any signal type other than HOLD/NULL is included in profit_factor (covers `STRONG_BUY`, `PAIR_TRADE`, `EXIT`, `COVER`, etc. if they ever ship).
- Cross-cutting AST guardrail: scan `views.py` for any SQL string containing `predicted_signal IN (` — fail the test, since this is the exact bug shape. Allowlist if a future query legitimately needs to target a closed set (force explicit decision).
- Risks: profit_factor will move on user-visible dashboards. Direction: usually toward more accurate (more trades counted, less luck of subset). On options profiles it could move dramatically (currently profit_factor reflects ~10% of trades; will reflect 100% post-fix). This is the correct direction — the displayed number was structurally wrong, not noisy.

**Outcome (2026-05-09):**
- ✅ Commit `c9598ff`. Both routes use `UPPER(predicted_signal) != 'HOLD'` (matching the existing `ai_tracker.py` convention). Silent `except: pass` upgraded to WARNING with failing db_path.
- ✅ Cross-cutting AST guardrail (`test_no_predicted_signal_in_whitelist_in_views_py`) blocks any `predicted_signal IN (...)` whitelist in `views.py`. Empty allowlist.
- ✅ 5 behavioral tests pin: every prod-real signal type included, HOLD case-insensitive exclusion, NULL signal exclusion, future-signal-type inclusion property, all-HOLD edge case.
- ✅ Live verify: web restarted 18:41:29 UTC. **0 exceptions on /performance or /ai since restart.** **0 profit_factor query warnings.**
- ✅ **Real prod numerical proof on largest options profile (id=3):**
  - OLD whitelist: profit_factor = 1.16 over 72 trades (BUY/SELL only)
  - NEW HOLD-exclude: profit_factor = **0.95 over 87 trades** (adds 15 SHORT+MULTILEG_OPEN trades)
  - The displayed value moved from 1.16 (ostensibly "profitable") to 0.95 (net losing) — the hidden SHORT/MULTILEG_OPEN trades have been net losers on this profile and were being silently hidden from the headline metric. **The user now sees the truth.**
